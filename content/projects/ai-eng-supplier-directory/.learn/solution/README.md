# Supplier Directory — Lightweight Storage API - Reference Solution

## Purpose

This reference solution explains how to implement the project end-to-end with production-like structure, strict validation, and a responsive UI experience.

Use this document as an implementation guide, not only as a checklist.

---

## 1) Solution Architecture

The expected implementation is split into two clear modules:

- `services/api`: FastAPI + Pydantic + TinyDB for persistence and domain validation.
- `uis/application`: UI that consumes the API, supports filtering, creation, and inline updates.

Key principle: all supplier rules from `CONTEXT-company.md` are source of truth.  
Do not hardcode generic values that do not match the assigned context.

---

## 2) Backend Requirements (`services/api`)

### 2.1 Data model and validation

Define a Pydantic model for suppliers with fields from the project context:

- Supplier name.
- Country.
- Product categories (list).
- Current rate.
- Last rate update timestamp.
- Status.

Implementation details to enforce:

- Status must be constrained to allowed values from context (prefer `Enum`).
- Rate must be strictly positive (`> 0`).
- Input and output schemas should be separated when fields are server-managed:
  - For example, `updated_at` should be generated by backend, not accepted from clients.

### 2.2 TinyDB setup

- Centralize DB initialization in a dedicated module (for example `database.py`).
- Keep table access in helper functions/repository layer if possible to avoid duplicated query logic.
- Ensure IDs returned by TinyDB are included in API responses.

### 2.3 Seeder (`seed.py`)

Expected behavior:

- Loads initial suppliers from the assigned company context.
- Executable with `uv run seed`.
- Idempotent: running multiple times does not duplicate existing suppliers.
- Prints inserted count in terminal at the end.

Suggested duplicate strategy:

- Build a natural unique key from context-specific fields (for example `name + country`) and check before insert.

### 2.4 Endpoints

Minimum behavior by endpoint:

- `POST /suppliers`
  - Validates payload with Pydantic.
  - Creates supplier and returns created record with ID.
  - Returns `422` for invalid payload.
- `GET /suppliers`
  - Returns all suppliers when no query params.
  - Supports `country` and `category` filtering.
- `GET /suppliers/{id}`
  - Returns supplier detail.
  - Returns `404` when not found.
- `PATCH /suppliers/{id}/rate`
  - Validates new positive rate.
  - Updates rate and writes current server timestamp to `updated_at`.
- `PATCH /suppliers/{id}/status`
  - Accepts only allowed statuses.
  - Returns `422` for invalid status.
- `DELETE /suppliers/{id}`
  - Deletes supplier by ID.
  - Returns `404` for unknown ID.

### 2.5 Error contract consistency

The API should keep consistent semantics:

- `404`: entity does not exist.
- `422`: payload or field-level validation fails.
- `200/201`: operation succeeds.

## 3) Frontend Requirements (`uis/application`)

### 3.1 Supplier directory page

The UI must include:

- A page reachable from application navigation/menu.
- Supplier list (table or cards) with main fields.
- Filter controls by country and category.
- Form to register suppliers.
- Row-level actions for rate and status updates.

### 3.2 UX states to include

At minimum, handle:

- Initial loading state.
- Empty result state (no suppliers).
- Submission/loading state for mutations.
- API error state with readable feedback.
- Visual distinction for active vs suspended suppliers (badge or style token).

### 3.3 PATCH updates from UI (inline editing)

For direct editing in interface, implement a safe mutation flow:

1. Enter edit mode in row (rate input or status selector).
2. Trigger PATCH request on confirm/save.
3. Disable row actions while request is in flight.
4. Update row data when response returns.
5. Exit edit mode and show success feedback.

Recommended approach for smooth UX: optimistic or semi-optimistic updates.

- Optimistic option:
  - Update row immediately in local state before API responds.
  - Mark row as "syncing".
  - If API fails, rollback previous value and show error.
- Conservative option:
  - Keep previous value visible while loading.
  - Replace value with server response only on success.

Important: for `PATCH /rate`, always trust backend `updated_at` from response and render that value in UI.

### 3.4 Suggested client state shape

Use explicit UI state to avoid race conditions:

- `suppliers: Supplier[]`
- `loadingList: boolean`
- `savingById: Record<number, boolean>`
- `errorById: Record<number, string | null>`
- `filters: { country?: string; category?: string }`

This allows independent row updates without blocking full list interactions.

## 4) Environment and Configuration

Even for this lightweight stack, the expected project setup should be consistent:

- Development environment ready to run backend and frontend.
- Test environment for automated endpoint and UI behavior checks.
- Production-like environment for final validation in a clean run.

Minimum configuration recommendations:

- Environment variables for API base URL in frontend.
- Separate data file path for TinyDB per environment when possible (dev/test).
- Scripted commands for:
  - seed
  - run backend
  - run frontend
  - tests

---

## 5) Quality and Testing Expectations

The solution should verify behavior, not only implementation.

Backend tests should cover:

- Valid supplier creation.
- Rejection of invalid status/rate (`422`).
- Country and category filtering logic.
- Not found behavior (`404`) for detail, patch, and delete.
- Rate patch updates `updated_at`.
- Seeder idempotency (second run does not duplicate).

Frontend checks should verify:

- List renders API data.
- Filters update visible results without full page reload.
- Create flow shows API validation errors correctly.
- Inline PATCH rate/status updates row correctly.
- Rollback path on failed PATCH when optimistic update is used.

---

## 6) Common Pitfalls to Avoid

- Using status values that do not match `CONTEXT-company.md`.
- Allowing zero/negative rates in any layer.
- Forgetting to update `updated_at` on rate change.
- Implementing filters only in frontend (must be API-backed).
- Replacing full list after each row patch without preserving local UI state.
- Seeder duplicating data on repeated execution.

---

## 7) Definition of Done

A complete solution is done when all below are true:

- All README checklist items are implemented.
- API routes behave exactly as specified.
- Seeder is runnable and idempotent.
- UI supports list, filters, create, and inline PATCH updates.
- Error/status handling is consistent and user-friendly.
- Submission evidence (screenshots required in project README) is prepared.
